数字IC验证

关于数字IC的一些验证原则、思路、方法等

第1章 什么是验证？

验证是一个证明设计思路是否正确实现的过程。

1.1 验证需要一个测试平台

在这个测试平台中，包含了设计组件。测试平台可以对设计组件进行驱动，给出合理的激励。 同时测试平台也可以接受设计组件给出的结果，以判断结果是否如预期的那样。 图 1.1 展示了这样一个测试平台。

1.2 验证的重要性

验证的工作量巨大，几乎所有的验证工作都不能及时完成，或多或少需要延期。对于重要功能进行完备的验证是必要的， 而对于一些不太重要的功能或者说使用频率较低的功能在紧急的情况下可以降低优先级，放在下一个版本中进行充分验证。

1.3 形式验证

形式验证不仅仅是简单的数学验证，可以从两个大类来理解形式验证：

  • 1.3.1 等效性检查
  • 1.3.2 模型检查
1.3.1 等效性检查

对两个网表进行比较。后一个网表相比于前一个网表进行了后处理。 后处理在插入扫描链、时钟树综合或修改网表不应该改变电路的功能。

1.3.2 模型检查

模型检查时形式验证奇数的一个新应用。它证明或否定了设计的断言或特征。例如，可以找到状态机是否存在不可到达的状态，是否存在死锁状态。

1.4 功能验证

功能验证是针对设计意图的。

功能验证的主要目的是确定一个设计是否实现了预期的功能。 将预期功能转化为代码形成激励注入和结果检查，以确定设计文档到RTL代码的转换是对应的。

功能验证可以有三种方式：

  • 1.4.1 黑盒验证 。
  • 1.4.2 白盒验证 。
  • 1.4.3 灰盒验证 。
1.4.1 黑盒验证

设计作为黑盒展示，不能检查或了解设计的内部细节。

针对黑河验证时，由于内部不可见性，不关心内部是如何实现的。 所有的验证都只能通过接口信号完成，包括激励的输入和结果的输出。

这种方法显然缺乏对设计的可控性和可观察性，对设计的了解止步于接口信号。

黑盒验证的优点是具有高度的一致性，无论内部设计是怎么样的，正确设计接口上的表现总是呈现为高度的一致性。

对于大规模设计，黑盒验证并不适用。设计内部的关键功能很难控制盒观测，只限于小规模的模块验证。

1.4.2 白盒验证

设计作为白盒展示，可以完全了解并控制设计的内部细节。

这种方法对于设计有完全的控制能力。在验证过程中，可以很清晰的看到内部寄存器的变化、状态机的跳转。 非常有利于验证人员协助设计人员进行问题定位。

1.4.3 灰盒验证

灰盒验证是介于黑河验证和白盒严验证之间的。

第2章 验证工具

2.1 lint工具

lint工具常用来检查代码的静态错误。包括端口未驱动、未连接、位宽不匹配等。

2.2 仿真工具

仿真器将预设的激励给到输入端口，并适时的进行设计内部状态机的跳转，寄存器和线网的变化，在合适的时机将输出呈现在输出端口上。

有些设计可能不至于一种设计语言，这就需要用到多种仿真器进行联合仿真。

2.3 波形浏览器

波形浏览器显示出随时间变化的信号值。

波形浏览器通常接收仿真器的一个输出文件。这个输出文件记载了设计内部信号随时间的变化情况。 有些波形文件还包含了设计的内容，可以方便的将信号在时间上的变化和设计代码对照，

第3章 问题追踪

验证工程师的工作就是已找出错误。如果没有发现错误，凡是一键值得怀疑的事情。 即使作为最有经验的设计师，也会存在错误，无论是复杂的功能错误还是简单的书写失误。

在验证过程中会出现两种错误：

  • 验证平台错误；
  • 设计代码错误；

对于验证平台错误，在初期搭建验证平台的时候最常见。在这一阶段，尽量不要给出复杂的激励，将重点放在验证平台的正确驱动激励和结果提取上。

对于设计代码错误，在验证平台已经很完善的时候，设计代码的错误在整体错误中的比例将逐步提升，但是也有可能出现一些验证平台的错误。 这时候要做的事就是仔细分析错误，确定是验证平台错误还是设计错误，只有大概率确定是后者错误的情况下才能反馈给设计。

3.1 值得追踪的问题

对所有出现的问题都进行追踪是不现实的。

要仔细分析每个出现的问题，进行合理的追踪。 值得追踪的问题，需要提交到bug list。

  • 如果问题不解决，某些重要功能就无法实现。
  • 如果问题不解决，验证平台就不能正确反映设计的验证过程。

不值得追踪的问题，可以忽略也可以作为经验list记录：

  • 驱动方法不对。
  • 组件连接有问题。
  • 设计人员确定问题优先级太低，可以放到下一个版本修复。

3.2 出现问题的讨论

口头讨论

对于一些不理解的、模糊的设计点可以直接与设计人员进行口头讨论。这也是使用最普遍的方法。

书面加口头讨论

针对相关人员不在同一办公地点的问题，应先列出一个清单，说明相关情况：

  • 问题描述。
  • 激励驱动。
  • 相关波形（截图或者列出相关信号）。
  • 仿真环境副本，便于相关人员直接在本地debug。
  • 可能的问题所在（这建立在验证人员对设计有一定的熟悉）。

在有了问题说明之后，可以先进行沟通，视情况组织会议共同讨论。

3.3 不能重复犯同样的错误

对于值得跟踪的问题，要做好详细的记录，确保不再犯同样的错误。 要从失败中汲取教训，避免因为同样的原因导致验证或设计出错。
